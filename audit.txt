## B. Critical Issues (fix before production)

### B1. Dual scheduler systems on the same data

Two backends both touch scheduling:

| System | Routes | Used by FE? |
|--------|--------|-------------|
| **Legacy** | `/api/schedule`, `GET /api/schedule/run` | Yes — `useGetSchedule`, `CreateSchedule`, `ScheduleList` |
| **QStash** | `/api/scheduler/*`, `/api/webhooks/qstash` | **No** — zero frontend calls |

Both read/write schedule-related JSON. Running both in prod risks duplicate sends, race conditions, and confusing ops.

**Recommendation:** Pick one (likely legacy until QStash is fully wired, or migrate FE to `/api/scheduler` and retire legacy).

### B2. Template approval not admin-gated

`approveTemplate` / `rejectTemplate` in `templateService.js` call the repo with no role check. Any authenticated user hitting `/api/document-templates/:id/approve|reject` can moderate community templates.

`PendingApprovalQueue.tsx` on the Community tab exposes approve/reject to all users.

**Recommendation:** Add `requireAdmin` on approve/reject routes; move queue UI to admin or hide actions for non-admins.

### B3. Broken route: `/dashboard/documents/create`

`DocumentTemplatesSection.tsx` links to `/dashboard/documents/create`, but only `fe/src/app/dashboard/documents/page.tsx` exists. Navigation 404s if that component is ever mounted.

## C. High Issues

### C1. Schedule History is mock data

`ScheduleHistory.tsx` uses hardcoded 2023 campaigns. `/dashboard/schedules/history` renders it with **no API**. Users see fake history.

### C2. `GET /api/schedule/run` requires auth

External cron cannot trigger runs without a JWT. Either intentional (manual only) or a design smell — needs a cron secret / internal route if automated runs are required.

### C3. JSON wallet/store shape inconsistency

`credits_wallets.json` was `{ __scoped: true, users: {} }` while admin code expected an array — patched via `readWalletList()` / `writeWalletList()`. Other global files may still have mixed shapes; not all code paths normalize.

### C4. Minimal test coverage

- `server/tests/production.test.js` — ~5 tests, mostly utils/placeholders  
- `fe/src/__tests__/production.test.ts` — placeholder logic, not real components  

No meaningful E2E or integration coverage for auth, billing, or send flows.

### C5. `SCHEDULER_ENABLED` / QStash env coupling

QStash module exists but is unused by FE. If enabled in prod without full wiring, it adds complexity with no user-visible benefit.

`dataConsistencyService.js` audits attachment references only; not run on startup or exposed in admin UI.

### D4. Admin moderation UX split

Community tab shows pending approvals to everyone; admin has no dedicated moderation queue (despite admin refactor elsewhere).

### D5. `AddAttachment` uses wrong fetch pattern

`useFetchCategory("/api/attachment")` — category hook against attachment API; dead code but signals copy-paste drift.

## I. Data & Persistence Risks

- **No migrations** — additive JSON changes only; shape drift handled ad hoc (`walletRepository`, `billingBootstrap.repairWalletStoreShape`)  
- **Scoped vs global files** — `__scoped: true` wrappers caused admin user list failure (fixed for wallets; pattern may repeat)  
- **Concurrent writes** — JSON file DB without locking beyond scheduler lock module (QStash path only)  
- **Credit expiry** — `startCreditExpiryScheduler()` on boot — verify idempotency in multi-instance deploy
